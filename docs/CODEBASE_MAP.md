## Architecture

# Architecture

**Analysis Date:** 2026-04-18

## Pattern Overview

**Overall:** Home Assistant custom integration using the standard `async_setup_entry` + `DataUpdateCoordinator` pattern with a cloud_push IoT class (WebSocket-driven state updates from Shelly Cloud Integrator API).

**Key Characteristics:**
- Single-instance integration (enforced via `async_set_unique_id(INTEGRATOR_TAG)` in `custom_components/shelly_integrator/config_flow.py`)
- Cloud-push data flow: JWT-authenticated WebSocket receives device state events; coordinator fans out to platform entities via HA dispatcher signals
- Layered separation: API (transport) → Coordinator (state orchestration) → Entities (HA platform objects) → Services (HA-facing actions)
- Gen1 (Block) and Gen2/Gen3 (RPC) devices handled through a single coordinator with `is_gen2_status()` branching at the entity layer
- Persistent device discovery: "known devices" persisted in `ConfigEntry.data["known_devices"]` so entities survive restart before WebSocket re-verifies them
- Webhook consent flow: Shelly Cloud posts to a registered HA webhook (`shelly_integrator_callback`) to add/remove devices post-OAuth

## Layers

- Purpose: Pure transport — JWT auth and WebSocket plumbing, no device/domain logic
- Location: `custom_components/shelly_integrator/api/`

## Data Flow

**Initial setup flow (config entry → operational entities):**

1. User enters integrator token in `custom_components/shelly_integrator/config_flow.py::async_step_user`
2. Config flow validates token by calling `API_GET_TOKEN` endpoint
3. User advances to `async_step_consent`, which renders consent URL (built by `core/consent.py::build_consent_url`) — user grants device access in Shelly Cloud
4. `custom_components/shelly_integrator/__init__.py::async_setup_entry` fires:
   - Constructs `ShellyAuth` and calls `get_jwt_token()` — raises `ConfigEntryNotReady` on failure
   - Constructs `ShellyIntegratorCoordinator` with JWT and entry
   - `coordinator._restore_known_devices()` reads `entry.data["known_devices"]` (device_id → host map) and seeds `self.devices[did] = {"online": False}` stubs
   - `async_config_entry_first_refresh()` schedules 23-hour `TOKEN_REFRESH_INTERVAL` and calls base-class refresh
   - Registers webhook `WEBHOOK_ID = "shelly_integrator_callback"`
   - `connect_to_host("shelly-187-eu.shelly.cloud")` kicks off WebSocket task
   - `async_wait_for_devices(timeout=5.0)` blocks until all pending device verifications complete (or timeout)
   - `_purge_deleted_entities()` removes ghost entity-registry records for this entry
   - `hass.config_entries.async_forward_entry_setups(entry, PLATFORMS)` sets up all 6 platforms
   - `NotificationService.show_setup_notification()` posts consent URL
   - `HistoricalDataService.setup_auto_sync()` schedules daily CSV sync if gateway URL configured

**Runtime event flow (WebSocket → entities):**

1. `ShellyWebSocket._connect_and_listen` opens `wss://{host}:6113/shelly/wss/hk_sock?t={jwt}` and fires `on_connected(host)` callback
2. `coordinator._on_ws_connected(host)` queues `Integrator:ActionRequest` with `action="DeviceVerify"` for every known device on that host, tracking them in `_pending_verifications`
3. Shelly Cloud replies with `Integrator:ActionResponse` events containing `deviceStatus`, `deviceSettings`, `accessGroups`
4. `coordinator._handle_ws_message` routes by `event` field:
   - `Integrator:ActionResponse` → `_handle_action_response` (DeviceVerify/DeviceGetSettings results, `WRONG_HOST` redirect, `UNAUTHORIZED`)
   - `Shelly:StatusOnChange` → `_handle_status_change` (partial status merge — preserves existing keys)
   - `Shelly:Online` → `_handle_online` (online/offline flag)
   - `Shelly:Settings` → `_handle_settings` (hostname, name)
5. On device addition/update, `async_set_updated_data(self.devices)` propagates to `CoordinatorEntity` subscribers; newly-discovered devices trigger `async_dispatcher_send(hass, SIGNAL_NEW_DEVICE, device_id)`
6. Each platform's `async_add_device(device_id)` callback builds entities from current `coordinator.devices[device_id]["status"]` structure and calls `async_add_entities(...)`

**Command flow (HA action → device):**

1. HA calls entity method (e.g., `ShellySwitch.async_turn_on`)
2. Entity calls `self.coordinator.send_command(device_id, cmd="relay", channel, action="on")` (or `send_jrpc_command` for JRPC RPC methods)
3. Coordinator looks up host from `_device_host_map` and delegates to `ShellyWebSocket.send_command`
4. WebSocket assigns a `trid`, registers a `Future` in `_pending_responses[trid]`, sends `Shelly:CommandRequest` JSON, awaits with timeout
5. Response (`Shelly:CommandResponse` or `Shelly:JrpcResponse`) is caught in `_handle_message` and resolves the future
6. Entity checks `_is_command_ok(response)` (handles `UNAUTHORIZED` JRPC error + Gen1 `isok=false`) and then `_update_local_state(...)` + `async_write_ha_state()` for optimistic UI update

**Webhook flow (Shelly Cloud consent → new device):**

1. User grants consent at `https://my.shelly.cloud/integrator.html`; Shelly Cloud POSTs one payload per device to `{ha_external_url}/api/webhook/shelly_integrator_callback`
2. HA webhook component invokes `WebhookHandler.handle_request`
3. `parse_webhook_payload` produces `WebhookPayload(device_id, action, host, name, device_type, device_code, access_groups)`
4. Branch on `action`:
   - `add` → `_handle_device_add` populates `coordinator._device_host_map` + `coordinator.devices`, connects to new host if needed, shows notification
   - `remove` → `coordinator.async_remove_device(device_id, remove_from_registry=True)`

**Historical sync flow (EM CSV → HA statistics):**

1. `HistoricalDataService._run_auto_sync` fires daily (or on service call `shelly_integrator.download_and_convert_history`)
2. `_find_em_devices` filters `coordinator.devices` by `device_code in {"SHEM", "SHEM-3", "SPEM-003CEBEU"}`
3. For each EM channel: `fetch_csv_from_gateway(gateway_url, hostname, channel)` → `parse_shelly_csv_for_import` → `(datetime_utc, delta_wh)` list
4. `_import_statistics_native` aligns cumulative sums against `statistics_during_period` recorder query, then calls `async_import_statistics` (upserts)
5. `_resolve_energy_entity_id` uses the entity registry to find current `statistic_id` by `unique_id = f"{device_id}_emeter|energy_{channel}"`

**State Management:**
- Runtime device state lives in `coordinator.devices: dict[str, Any]` — keyed by Shelly Cloud `deviceId`, each value is `{"status": dict, "device_type": str, "device_code": str, "name": str, "access_groups": str, "online": bool, "settings": dict}`
- `coordinator._device_host_map: dict[str, str]` tracks which WebSocket server owns each device
- Persistence: `_persist_known_devices` writes `{device_id: host}` map into `ConfigEntry.data["known_devices"]` via `hass.config_entries.async_update_entry`
- Entity state derives purely from `coordinator.devices[self._device_id]["status"]` via `ShellyBaseEntity.device_status` property — no local entity state
- Optimistic updates: switch/light/cover mutate `status` dict in place, then call `async_write_ha_state()` (actual WebSocket push arrives later as `Shelly:StatusOnChange`)

## Error Handling

**Strategy:** Fail-soft at the edges, fail-loud on setup. Network/WebSocket errors are logged and retried with exponential backoff; setup-time failures raise `ConfigEntryNotReady` so HA schedules a retry.

**Patterns:**
- **Setup failure:** `async_setup_entry` wraps JWT acquisition in try/except and re-raises as `ConfigEntryNotReady(...)` (`custom_components/shelly_integrator/__init__.py:56-59`)
- **WebSocket reconnection:** `ShellyWebSocket._connection_loop` uses exponential backoff bounded by `WS_RECONNECT_MIN=1` → `WS_RECONNECT_MAX=60` seconds; backoff resets on a successful session (`custom_components/shelly_integrator/api/websocket.py:285-301`)
- **Command responses:** `send_command` / `send_jrpc_request` swallow `asyncio.TimeoutError` and generic exceptions, returning `None` so callers can branch on a missing response (`custom_components/shelly_integrator/api/websocket.py:171-183`)
- **UNAUTHORIZED diagnostics:** `coordinator._validate_device_access` emits `ERROR`-level diagnostics (accessGroups bitmask, online state, device type) to aid user troubleshooting (`custom_components/shelly_integrator/coordinator.py:261-326`)
- **Webhook errors:** `WebhookHandler.handle_request` returns HTTP 400 on invalid payload, 500 on unexpected exception, 200 `OK` otherwise (`custom_components/shelly_integrator/services/webhook.py:40-66`)
- **Historical sync:** `_import_statistics_native` catches all exceptions, logs, and returns `False`; `_run_auto_sync` catches to prevent auto-sync task death (`custom_components/shelly_integrator/services/historical.py:289-291`, `:123-134`)
- **Token refresh failure:** `coordinator._async_refresh_token` logs and continues — existing WebSocket remains until its next natural drop (`custom_components/shelly_integrator/coordinator.py:208-214`)

## Cross-Cutting Concerns

**Logging:** Module-level `_LOGGER = logging.getLogger(__name__)` in every file; ERROR for user-visible failures, WARNING for transient issues, INFO for lifecycle events, DEBUG for message payloads.

**Validation:** `voluptuous` schemas only at HA boundaries — config flow (`STEP_USER_DATA_SCHEMA`) and service registration (`shelly_integrator.download_and_convert_history` schema in `__init__.py::_register_services`).

**Authentication:** Single `ShellyAuth` instance owns JWT state; coordinator passes a `lambda: self._auth.jwt_token` provider to `ShellyWebSocket` so reconnects always pick up the latest token. `TOKEN_REFRESH_INTERVAL = 23 * 60 * 60` (23 hours) schedules proactive refresh before the 24-hour expiry.

**Concurrency:** Fully async (`aiohttp`, `asyncio`). `ShellyWebSocket` tracks outstanding background tasks in `self._tasks`; `disconnect_all` cancels them cleanly. Historical sync uses `hass.async_add_executor_job` for synchronous recorder queries to avoid blocking the event loop (`services/historical.py:150`).

**Persistence:** Device-to-host map persisted via `hass.config_entries.async_update_entry(entry, data={..., "known_devices": {...}})`. No custom storage — leverages `ConfigEntry.data`. Entity states persist through the HA recorder; energy statistics go through the HA statistics API (`async_import_statistics`).

**Entity naming:** `ShellyBaseEntity._get_device_name` appends `({device_id})` to every device name to guarantee unique entity IDs even when users duplicate names in Shelly Cloud (`custom_components/shelly_integrator/entities/base.py:85-134`).

## APIs & External Services

**Shelly Cloud Integrator API (primary integration target):**

- **Token endpoint (HTTPS REST):** `https://api.shelly.cloud/integrator/get_access_token`
  - Method: `POST` with form data `{itg: INTEGRATOR_TAG, token: <user_token>}`
  - Returns JWT in `data.data` field; wrapper signals success via `data.isok` boolean
  - Client: `custom_components/shelly_integrator/api/auth.py` (`ShellyAuth.get_jwt_token`)
  - Also called during config flow validation: `custom_components/shelly_integrator/config_flow.py:64-82` (user step), `:134-154` (reconfigure), `:198-207` (options)
  - SDK/Client: raw `aiohttp.ClientSession.post` (no SDK used)
  - Auth: `INTEGRATOR_TAG = "ITG_OSS"` (non-secret) + user-supplied integrator token (secret)

- **Shelly Cloud WebSocket servers (WSS):** `wss://{host}:6113/shelly/wss/hk_sock?t={jwt_token}`
  - Default host: `shelly-187-eu.shelly.cloud` (EU; `DEFAULT_SERVER` in `custom_components/shelly_integrator/__init__.py:43`)
  - Additional hosts discovered dynamically via `WRONG_HOST` response and webhook payloads (e.g., `shelly-*-eu.shelly.cloud`, other regional servers)
  - Client: `custom_components/shelly_integrator/api/websocket.py` (`ShellyWebSocket`) using `aiohttp.ClientWebSocketResponse`
  - Protocol: bespoke JSON-over-WebSocket framing (not standard JSON-RPC at transport level):
    - Outbound envelopes — `event`, `trid`, `deviceId`, `data`/`method`/`params`:
      - `Shelly:CommandRequest` — Gen1 relay/light/roller commands (`websocket.py:112-183`)
      - `Shelly:JrpcRequest` — Gen2/Gen3 RPC methods (Switch.Set, Light.Set, Cover.Open, etc.) (`websocket.py:185-252`)
      - `Integrator:ActionRequest` — `DeviceVerify`, `DeviceGetSettings` (`websocket.py:254-283`)
    - Inbound events (routed in `custom_components/shelly_integrator/coordinator.py:372-390`):
      - `Shelly:CommandResponse` / `Shelly:JrpcResponse` — correlated by `trid`, resolved via `asyncio.Future`
      - `Integrator:ActionResponse` — device verification results including `OK`, `WRONG_HOST` (with redirect host), `UNAUTHORIZED`
      - `Shelly:StatusOnChange` — partial status updates (shallow-merged into existing device status)
      - `Shelly:Online` — online/offline transitions
      - `Shelly:Settings` — device settings push
      - `Error` — server-side errors
  - Reconnect: exponential backoff 1 s → 60 s (`WS_RECONNECT_MIN`/`WS_RECONNECT_MAX` in `const.py:28-29`, logic at `websocket.py:285-301`)
  - Auth: JWT passed as `?t=` query parameter on the wss URL (`websocket.py:306`)
  - TLS: `ssl=True` explicit (`websocket.py:310`)

## Error Handling

**Primary Pattern — Log and Continue (library code):**
All optional / network-facing operations wrap in `try` / `except Exception as err:` and log, returning `None` / empty result instead of propagating.
```python
# custom_components/shelly_integrator/api/websocket.py:171-183
try:
    await ws.send_json(command)
    result = await asyncio.wait_for(future, timeout=timeout)
    _LOGGER.debug("Command response: %s", result)
    return result
except asyncio.TimeoutError:
    _LOGGER.warning("Command timeout for %s", device_id)
    self._pending_responses.pop(trid, None)
    return None
except Exception as err:
    _LOGGER.error("Command error: %s", err)
    self._pending_responses.pop(trid, None)
    return None
```

**Setup Pattern — Convert to HA Ready Errors:**
`async_setup_entry` raises `ConfigEntryNotReady` when required setup work fails, using exception chaining with `from err`.
```python
# custom_components/shelly_integrator/__init__.py:56-59
try:
    jwt_token = await auth.get_jwt_token()
except Exception as err:
    raise ConfigEntryNotReady(f"Failed to get JWT token: {err}") from err
```

**Config Flow Pattern — Map to User-Friendly error keys:**
Config flow catches specific exceptions, maps to `errors["base"]`, uses `_LOGGER.exception("Unexpected exception")` for the unknown branch.
```python
# custom_components/shelly_integrator/config_flow.py:83-87
except aiohttp.ClientError:
    errors["base"] = "cannot_connect"
except Exception:
    _LOGGER.exception("Unexpected exception")
    errors["base"] = "unknown"
```

**Narrow-then-Broad Order:**
Handlers catch the specific exceptions first (`asyncio.TimeoutError`, `aiohttp.ClientError`, `json.JSONDecodeError`, `ValueError`, `IndexError`) then fall back to a broad `except Exception as err` that logs at `error` or `exception` level — see `utils/http.py:47-52`, `utils/csv_converter.py:55-57`.

**Raising Custom Errors:**
- `ShellyAuth.get_jwt_token` raises `ValueError` on API-level error (`api/auth.py:71`).
- `HomeAssistantError` is **not currently used** — failures bubble up as `ConfigEntryNotReady` from setup, or are swallowed with logging in background tasks. When adding new user-facing service errors, prefer raising `homeassistant.exceptions.HomeAssistantError` (import pattern consistent with HA conventions).

**Background Task Errors:**
Connection loops treat exceptions as reconnect triggers rather than propagating — see `api/websocket.py:285-301`.

- Issue: The POST-to-`API_GET_TOKEN`/parse-`isok` pattern is open-coded in three places instead of using `ShellyAuth.validate_token()`

## Known Bugs

**Ghost entity race window with entity-registry reload:**
- Symptoms: Re-adding the same device after deletion can still yield old entity IDs if the registry save from `_purge_device_entities` has not been flushed before the user clicks "Add" again
- Files: `custom_components/shelly_integrator/__init__.py:138-219`
- Trigger: Rapid delete-then-reconfigure cycle before HA saves `core.entity_registry`
- Workaround: Startup purge (`_purge_deleted_entities`) cleans it on next restart. Commit history shows this has been iterated on (`aa85064`, `2aad50d`) and remains a best-effort mitigation.

**Command-response `trid` collisions across concurrent JRPC calls:**
- Symptoms: `_message_id` is a shared counter incremented without a lock; if a JRPC and a CommandRequest start in overlapping tasks the same `trid` can theoretically be assigned twice because the increment is not atomic from the event-loop's perspective when awaited across `await ws.send_json(...)` boundaries
- Files: `custom_components/shelly_integrator/api/websocket.py:47`, `:141-142`, `:213-214`, `:272-273`
- Trigger: Two simultaneous HA service calls hitting the same coordinator
- Workaround: In practice Python asyncio makes the increment atomic between awaits, but the pattern is fragile. Use `itertools.count()` or an explicit lock.

**EM channel count hardcoded by device code:**
- Symptoms: Only `SHEM-3` and `SPEM-003CEBEU` get 3 channels; all other EM devices default to 2. Any new 3EM variant Shelly releases will miss channel 2.
- Files: `custom_components/shelly_integrator/services/historical.py:336-340`, `:37`
- Trigger: User adds a newer EM model not in `EM_DEVICE_CODES`
- Workaround: Inspect `status.emeters` length at runtime

**Brightness conversion loses precision at 0:**
- Symptoms: `brightness` rounds via `int(pct * 255 / 100)` and reverse; setting HA brightness to 1 can result in 0% at Shelly, which some firmware treats as "off"
- Files: `custom_components/shelly_integrator/light.py:140`, `:170`
- Trigger: Dimming to very low values
- Workaround: None; round-trip conversion is lossy by design

**DeviceGetSettings response can overwrite settings with stub:**
- Symptoms: In `_handle_action_response`, when a DeviceGetSettings response has `device_status = {}`, the `else` branch only copies name/code/type into `existing`, but if `existing` had a live `status` from `StatusOnChange` that arrived first, the assignment `self.devices[device_id] = existing` is still safe — however `settings` acquired out-of-band is lost when DeviceVerify later fires (only `prev_settings` is preserved, not nested fields)
- Files: `custom_components/shelly_integrator/coordinator.py:414-478`
- Trigger: Specific ordering of StatusOnChange → Shelly:Settings → DeviceVerify
- Workaround: Rarely manifests because DeviceVerify typically fires before Shelly:Settings arrives

## Security Considerations

**Webhook has no authentication beyond a guessable ID:**
- Risk: `WEBHOOK_ID = "shelly_integrator_callback"` is a fixed string baked into `const.py`. Anyone who can reach the HA external URL can POST to `/api/webhook/shelly_integrator_callback` and trigger `_handle_device_add` / `_handle_device_remove`, causing spurious devices or silent removal of real ones.
- Files: `custom_components/shelly_integrator/const.py:45`, `custom_components/shelly_integrator/services/webhook.py:40-66`, `custom_components/shelly_integrator/__init__.py:75-82`
- Current mitigation: None beyond the need for HTTPS external reachability; no HMAC, no shared secret, no `userId` validation in `parse_webhook_payload`
- Recommendations: (1) Randomize `WEBHOOK_ID` per-install and store in config entry data. (2) Verify `userId` field in payload matches an expected account-level identifier. (3) Consider Shelly Cloud's signing convention if/when available.

**JWT token embedded in WebSocket query string:**
- Risk: JWT passed as `?t={jwt_token}` — URLs can leak via proxy/access logs, crash dumps, or browser history (for debugging tools). JWT grants full integrator control for up to 24h.
- Files: `custom_components/shelly_integrator/api/websocket.py:306`
- Current mitigation: `ssl=True` on the WSS connection, token refreshed every 23h
- Recommendations: This is a Shelly Cloud API constraint (they require the query-string auth). Mitigate by ensuring `_LOGGER.debug` never logs the full URL — currently line 308 logs only `host`, which is correct. Add a comment warning future contributors not to log `url`.

**JWT token stored in plaintext config-entry memory:**
- Risk: `self._jwt_token` held in `ShellyAuth`; `CONF_INTEGRATOR_TOKEN` stored in `entry.data` (HA-managed storage, plaintext on disk). Anyone with filesystem access to `.storage/core.config_entries` gets the integrator token, which yields full cloud integrator control.
- Files: `custom_components/shelly_integrator/api/auth.py:40-41`, `custom_components/shelly_integrator/__init__.py:51`
- Current mitigation: Standard HA config-entry storage (not encrypted)
- Recommendations: This is standard HA practice; cannot be fundamentally fixed here. Document in README that the token grants full device control.

**Consent URL exposes HA external URL to Shelly Cloud:**
- Risk: `build_consent_url` embeds the user's HA external URL in the `cb` query param, sent to `my.shelly.cloud`. This leaks the HA instance's public address to Shelly Cloud's logs/analytics.
- Files: `custom_components/shelly_integrator/core/consent.py:29-46`
- Current mitigation: Inherent to the consent protocol (Shelly needs the callback URL)
- Recommendations: None — required by design. Document in README.

**Webhook handler catches all exceptions and returns 500 without structured logging:**
- Risk: `except Exception as err: _LOGGER.error("Webhook error: %s", err)` — loses stack trace, `err` could echo payload content including device IDs
- Files: `custom_components/shelly_integrator/services/webhook.py:64-66`
- Current mitigation: Returns 500 on any failure
- Recommendations: Use `_LOGGER.exception(...)` to preserve tracebacks; sanitize the error message

**No URL validation on `local_gateway_url`:**
- Risk: User-supplied string concatenated directly into request URL: `f"{gateway_url}/{hostname}/emeter/{channel}/em_data.csv"`. A malicious (or misconfigured) gateway URL could SSRF the HA host.
- Files: `custom_components/shelly_integrator/utils/http.py:35`, `custom_components/shelly_integrator/config_flow.py:32`
- Current mitigation: Field is optional; only used for an outbound GET from HA
- Recommendations: Validate scheme (http/https only) and reject localhost/private-IP targets unless explicitly opted in. Use `vol.Url` in schema.

## Performance Bottlenecks

**Historical sync runs in a single serialized loop:**
- Problem: `sync_data` iterates EM devices × channels sequentially; each `fetch_csv_from_gateway` has a 120-second timeout. With N devices × 3 channels, worst-case stall is `N * 3 * 120s`.
- Files: `custom_components/shelly_integrator/services/historical.py:299-379`
- Cause: No concurrency between channel fetches
- Improvement path: Use `asyncio.gather` over `(dev_id, channel)` pairs with a bounded `Semaphore(5)` to avoid overloading the local gateway

**Initial historical sync blocks startup notification chain:**
- Problem: `async_at_started` fires `_on_ha_started`, which creates a task, but `setup_auto_sync` is awaited inline during `async_setup_entry`; if HA is already started by the time the integration loads (reload scenario), the callback fires immediately and the task races with platform setup
- Files: `custom_components/shelly_integrator/services/historical.py:91-121`, `custom_components/shelly_integrator/__init__.py:114-115`
- Cause: Kick-off is non-awaited but its recorder I/O can contend with the recorder's startup batch
- Improvement path: Delay first sync by `delay(60s)` on reload paths, or gate on `hass.is_running`

**`statistics_during_period` on every import call:**
- Problem: For each channel, `_get_recorder_sum` hits the recorder DB via `async_add_executor_job` with a 48-hour window just to read the last sum
- Files: `custom_components/shelly_integrator/services/historical.py:136-175`, `:232-250`
- Cause: No caching between channels of the same device
- Improvement path: Batch-query all statistic_ids for a device in a single `statistics_during_period` call

**WebSocket reconnect backoff caps at 60s:**
- Problem: After a long Shelly Cloud outage, every HA instance reconnects within 60s — thundering herd
- Files: `custom_components/shelly_integrator/const.py:29`, `custom_components/shelly_integrator/api/websocket.py:285-301`
- Cause: `WS_RECONNECT_MAX = 60`, no jitter
- Improvement path: Add jitter: `await asyncio.sleep(backoff + random.uniform(0, backoff * 0.1))`

**Fallback polling at 30s interval is dead code with no fallback value:**
- Problem: `DataUpdateCoordinator(update_interval=timedelta(seconds=30))` drives `_async_update_data` which just returns `self.devices` — no fetch occurs. The timer fires every 30s for nothing.
- Files: `custom_components/shelly_integrator/coordinator.py:64`, `:159-161`
- Cause: Leftover from a polling-based design; now purely push-driven via WebSocket
- Improvement path: Set `update_interval=None` to disable the timer

## Fragile Areas

**Device removal has three entry points with slightly different semantics:**
- Files: `custom_components/shelly_integrator/__init__.py:280-318` (`async_remove_config_entry_device`), `custom_components/shelly_integrator/coordinator.py:576-606` (`async_remove_device`), `custom_components/shelly_integrator/services/webhook.py:109-118` (`_handle_device_remove`)
- Why fragile: UI delete path purges entities + ghost records synchronously; webhook remove path calls `async_remove_device(..., remove_from_registry=True)` which does NOT purge ghost records. A webhook-driven remove followed by a re-add leaves ghost entity IDs.
- Safe modification: When adding a new removal trigger, route through a single method that does both entity purge and coordinator cleanup
- Test coverage: No test suite exists in the repository

**Entity creation tracks `created_entities` in a per-platform closure:**
- Files: `custom_components/shelly_integrator/switch.py:28-72`, `custom_components/shelly_integrator/light.py:32-76`, `custom_components/shelly_integrator/cover.py:32-74`, `custom_components/shelly_integrator/sensor.py:34-82`, `custom_components/shelly_integrator/binary_sensor.py:34-80`, `custom_components/shelly_integrator/button.py:35-86`
- Why fragile: The `created_entities` set lives in the platform's `async_setup_entry` closure; if HA reloads the platform without reloading the integration the set is re-initialized, leading to duplicate-entity attempts (unique_id collisions) unless HA's own deduplication kicks in
- Safe modification: Never call `async_forward_entry_setups` twice without full unload; move tracking into coordinator
- Test coverage: None

**Initial device-verification race requires 5s timeout:**
- Files: `custom_components/shelly_integrator/__init__.py:90`, `custom_components/shelly_integrator/coordinator.py:163-181`
- Why fragile: Commit `ed573b6 "Fix race condition: wait for device verification before platform setup"` shows this was an actual bug. The 5s budget is arbitrary — on slow cloud responses devices show up as `online=False` stubs and their platforms create zero entities until the next StatusOnChange.
- Safe modification: If increasing the platform list, budget more time per device (current fixed 5s)
- Test coverage: None

**Status merge assumes shallow keys:**
- Files: `custom_components/shelly_integrator/coordinator.py:505-536`
- Why fragile: `merged_status = {**existing_status, **new_status}` is a shallow merge. A StatusOnChange delivering `{"switch:0": {"output": true}}` replaces the ENTIRE `switch:0` object, losing `apower`/`voltage`/`current` from the previous full status. Sensors read `None` until the next full status.
- Safe modification: Deep-merge nested dicts, or re-request DeviceVerify after every StatusOnChange that looks partial
- Test coverage: None

**Consent URL building silently falls back on missing external URL:**
- Files: `custom_components/shelly_integrator/config_flow.py:250-257`, `custom_components/shelly_integrator/__init__.py:99-107`
- Why fragile: If `get_url(hass, prefer_external=True)` fails, user gets a consent URL with no `cb=` — the webhook never fires and devices never appear, with only a `warning`-level log
- Safe modification: Surface this as a persistent notification or config-flow error, not just a warning

## Scaling Limits

**Single global WebSocket-message ID counter:**
- Current capacity: `self._message_id` is an int, so effectively unlimited, but `_pending_responses` dict grows with every timed-out request until the process restarts if the cleanup path in `except` is missed (the current code does pop on timeout and error — good)
- Limit: A malicious server could refuse to respond and slowly grow memory via `Future` objects; bounded only by the 10s command timeout × QPS
- Scaling path: Add a periodic sweep of `_pending_responses` for abandoned entries

**Per-device status requests on WebSocket connect:**
- Current capacity: Every `_on_ws_connected` issues one `DeviceVerify` per known device on that host, then waits on `_devices_ready`
- Limit: With 100+ devices on one host, startup sends 100 messages back-to-back with no flow control
- Scaling path: Batch DeviceVerify or chunk with small delays

**`CONF_KNOWN_DEVICES` stored in config entry data:**

## Missing Critical Features

**No rate-limit / backoff on token refresh failures:**
- Problem: If `_async_refresh_token` fails, `_token_refresh_unsub` still fires every 23h (no retry, no shorter-interval retry on failure). Users with a revoked token have one failed reconnection then wait a full day.
- Blocks: Early recovery from transient Shelly Cloud 5xx

**No device-level diagnostics panel:**
- Problem: HA has `diagnostics` platform; this integration does not implement `async_get_config_entry_diagnostics` or `async_get_device_diagnostics`
- Blocks: Users cannot export a sanitized snapshot to share in issue reports; all debugging is log-trawling

**No `DataUpdateCoordinator.async_refresh()` path from user action:**
- Problem: There is no "reload now" button/service. Users must restart HA to force a reconnect.
- Blocks: Recovery from silent WebSocket half-open states

**No observability for consent-webhook delivery:**
- Problem: If a user clicks "Grant Access" but the webhook never fires (firewall, wrong external URL), there is no feedback in HA
- Blocks: Self-service troubleshooting

**Name field sometimes a string, sometimes a list:**
- Problem: Webhook payload has `name` as array (`data.get("name", [])[0]`); Shelly:Settings has `name` as string; DeviceVerify settings may nest under `device.hostname`. Priority list in `_get_device_hostname` handles it but is fragile.
- Files: `custom_components/shelly_integrator/core/consent.py:81-83`, `custom_components/shelly_integrator/services/historical.py:420-444`
- Blocks: Future Shelly API revisions silently break hostname resolution
